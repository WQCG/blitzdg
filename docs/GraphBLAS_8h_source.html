162  // The GraphBLAS spec lists this as an 'error' code, but it just means that
163  // A(i,j) is not present in the matrix, having been requested by
164  // GrB_*_extractElement. The function cannot return the proper value
165  // because the value of 'implicit zeros' depends on the semiring. For the
166  // conventational plus-times semiring, the implied 'zero' actually has the
167  // value of zero. For the max-plus semiring, it has the value -infinity.
168  // A matrix does not keep track of its semiring, and the user can change
169  // the semiring used to operate on the matrix. How mathematically
170  // well-defined that change of semiring is depends the user; GraphBLAS will
171  // not change the explicit values in the matrix if the semiring changes.
172  // As a result, GraphBLAS needs to return not a value, but an indication
173  // that the value of A(i,j) is implicit. The user application can use this
174  // indicator (GrB_NO_VALUE) to use the semiring's addititive identity, or
175  // it can take other action, as it chooses. In either case, it is safe to
176  // ask for values that are not there, which is why this return condition is
177  // not really an 'error' code but an informational code.

217 // GrB_init must called before any other GraphBLAS operation. GrB_finalize
218 // must be called as the last GraphBLAS operation.
219 
220 // GrB_init defines the mode that GraphBLAS will use: blocking or
221 // non-blocking. With blocking mode, all operations finish before returning to
222 // the user application. With non-blocking mode, operations can be left
223 // pending, and are computed only when needed.
224 
225 // The GrB_wait ( ) function forces all pending operations to complete.
226 // Blocking mode is as if GrB_wait is called whenever a GraphBLAS method or
227 // operation returns to the user.
228 
229 // The non-blocking mode is unpredictable if user-defined functions have side
230 // effects or if they rely on global variables, which are not under the control
231 // of GraphBLAS. Suppose the user application creates a user-defined operator
232 // that accesses a global variable. That operator is then used in a GraphBLAS
233 // operation, which is left pending. If the user application then changes the
234 // global variable before pending operations complete, the pending operations
235 // will be eventually computed with this different value.
236 
237 // The non-blocking mode can have side effects if user-defined functions have
238 // side effects or if they rely on global variables, which are not under the
239 // control of GraphBLAS. Suppose the user creates a user-defined operator that
240 // accesses a global variable. That operator is then used in a GraphBLAS
241 // operation, which is left pending. If the user then changes the global
242 // variable before pending operations complete, the pending operations will be
243 // eventually computed with this different value.
244 
245 // Worse yet, a user-defined operator might be freed before it is needed to
246 // finish a pending operation. This causes undefined behavior. To avoid this,
247 // call GrB_wait before modifying any global variables relied upon by
248 // user-defined operators, or before freeing any user-defined types, operators,
249 // monoids, or semirings.

253  GrB_NONBLOCKING, // methods may return with pending computations

255 }

544 // There are three sets of built-in binary operators. For the first set of
545 // 17 kinds of operators, x,y,z all have the same type, and they are available
546 // for all 11 types, for a total of 17*11 = 187 operators. All of them have
547 // a "_TYPE" suffix that denotes the type of x,y,z:
548

550 // 6 comparison: ISEQ, ISNE, ISGT, ISLT, ISGE, ISLE
551 // 3 logical: LOR, LAND, LXOR

554 // same type but z is always boolean, for a total of 6*11 = 66 operators.
555 // All of them have a "_TYPE" suffix that denotes the type of x,y (not z):
556 
557 // 6 comparison: EQ, NE, GT, LT, GE, LE
558 
559 // The final set of operators is for boolean x,y,z only, and they have no
560 // suffix:
561 
562 // 3 logical: LOR, LAND, LXOR
563

565 // but are included to keep the name space of operators uniform.

568 // FIRST, SECOND, MIN, MAX, PLUS, MINUS, TIMES, and DIV, for all 11 types.

713 // SPEC: the definition of divide-by-zero is an extension to the spec
714

716 // not define what boolean "division" means. SuiteSparse:GraphBLAS makes the
717 // following interpretation.
718 
719 // GraphBLAS does not generate exceptions for divide-by-zero, so the results
720 // are defined just as they are in MATLAB. Floating-point divide-by-zero
721 // follows the IEEE 754 standard: 1/0 is +Inf, -1/0 is -Inf, and 0/0 is NaN.
722 // For integer division by zero, if x is positive, x/0 is the largest integer,
723 // -x/0 is the integer minimum (zero for unsigned integers), and 0/0 is zero.
724 // For example, for int8, 1/0 is 127, and -1/0 is -128. For uint8, 1/0 is 255
725 // and 0/0 is zero.
726 
727 // Boolean division is not in MATLAB. For SuiteSparse:GraphBLAS, boolean
728 // division is treated as if it were an unsigned integer type with true=1 and
729 // false=0, and with the max and min value being 1 and 0. As a result,
730 // GrB_IDENTITY_BOOL, GrB_AINV_BOOL, and GrB_MINV_BOOL all give the same result

1289 // GrB_Vector_build: w = sparse (I,1,X) in MATLAB notation, but using any
1290 // associative operator to assemble duplicate entries.
1291 
1292 // Build a vector w from a set of (i,x) tuples. The type and dimension of the
1293 // vector is already defined in w (via GrB_Vector_new), which must initially
1294 // have no entries. I [0..nvals-1] is the list of row indices, and X
1295 // [0..nvals-1] is the list of numerical values. The kth tuple is (I[k],X[k]),
1296 // and tuples can appear in any order. Values are typecasted from X into the
1297 // type of the dup operator, as needed (user-defined types cannot be cast).
1298 // Duplicates are assembled together with the dup operator. If two tuples
1299 // (i,x1) and (i,x2) have the same row index, then w(i) = dup (x1,x2). All
1300 // three types of dup must be the same. The types of C, X, and dup must be
1301 // compatible.
1302 
1303 // SPEC: extension: well-defined behavior of a non-associative dup operator.
1304 
1305 // The GraphBLAS spec requires dup to be associative and does not define the
1306 // order in which duplicates are assembled. Currently this implementation
1307 // assembles duplicates in the order they appear in I and X. For example, if
1308 // (i,x1), (i,x2), and (i,x3) appear in that order in I and X, then w(i) =
1309 // dup(dup(x1,x2),x3). This means that using the non-associative FIRST
1310 // operator as dup means that w(i) is set equal to the first entry in the list,
1311 // x1, and SECOND gives the last one, x3. SuiteSparse:GraphBLAS guarantees
1312 // this ordering. However, per the spec, this order of assembly is not
1313 // guaranteed in all implementations. Thus dup must be associative and results
1314 // are not guaranteed in all implementations if it is not.

2591 // The Descriptor is used to modify the behavior of GraphBLAS operations.
2592 //
2593 // OUTP: can be DEFAULT or REPLACE. If REPLACE, then C is cleared after taking
2594 // part in the accum operation but before the mask. In other words,
2595 // C<Mask> = accum (C,T) is split into Z = accum(C,T) ; C=0 ; C<Mask> = Z.
2596 //
2597 // MASK: can be DEFAULT or SCMP. If DEFAULT, the mask is used normally,
2598 // where Mask(i,j)=1 means C(i,j) can be modified by C<Mask>=Z, and
2599 // Mask(i,j)=0 means it cannot be modified even if Z(i,j) is has been
2600 // computed and differs from C(i,j). If SCMP, this is the same as
2601 // taking the logical complement of the Mask.
2602 //
2603 // INP0: can be DEFAULT or TRAN. If DEFAULT, the first input is used as-is.
2604 // If TRAN, it is transposed. Only matrices are transposed this way.
2605 //
2606 // INP1: the same as INP0 but for the second input
2607

2609 {
2610  GrB_OUTP, // descriptor for output of a method
2611  GrB_MASK, // descriptor for the mask input of a method
2612  GrB_INP0, // descriptor for the first input of a method
2613  GrB_INP1 // descriptor for the second input of a method
2614 }

3092 // SPEC: The GxB_*_subassign functions are extensions to the spec.
3093 
3094 // Each GxB_subassign function is very similar to its corresponding GrB_assign
3095 // function in the spec, but they differ in three ways:
3096 
3097 // (1) the mask in the GxB_subassign functions has the same dimensions as
3098 // w(I) for vectors and C(I,J) for matrices. In GrB_assign, the mask is
3099 // the same size as w or C, respectively (except for GrB_Row_asssign and
3100 // GrB_Col_assign, in which case the mask is the same size as a row or
3101 // column of C, respectively). The two masks are related. If M is the
3102 // mask for GrB_assign, then M(I,J) is the mask for GxB_subassign. If
3103 // there is no mask, or if I and J are both GrB_ALL, then the two masks
3104 // are the same.
3105 
3106 // For GrB_Row_assign and GrB_Col_assign, the mask vector is the same
3107 // size as a row or column of C, respectively. For the corresponding
3108 // GxB_Row_subassign and GxB_Col_subassign operations, the mask is the
3109 // same size as the subrow C(i,J) or subcolumn C(I,j), respectively.
3110 
3111 // (2) They differ in how C is affected in areas outside the C(I,J) submatrix.
3112 // In GxB_subassign, C(I,J) is the only part of C that can be modified,
3113 // and no part of C outside the submatrix is ever modified. In
3114 // GrB_assign, it is possible to modify C outside the submatrix, but only
3115 // in one specific manner. Suppose the mask M is present (or, suppose it
3116 // is not present but GrB_SCMP is true). After (optionally) complementing
3117 // the mask, the value of M(i,j) can be 0 for some entry outside the
3118 // C(I,J) submatrix. If the GrB_REPLACE descriptor is true, the
3119 // GrB_assign deletes this entry. This case does not occur if GrB_REPLACE
3120 // is false. With GrB_assign, it is not possible to change entries
3121 // outside the submatrix C(I,J), except to delete them in this

3146 // then use GrB_assign. If the Mask is meant to control which entries of the

3148 // GxB_subassign. This is particularly useful is the Mask is a "template" that
3149 // follows along with the finite-element F, independent of where it is applied
